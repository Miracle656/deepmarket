### Upgrading with the current CLI (1.72.1)

The toolchain that shipped this build changed how publish addresses are
tracked, and is older than current testnet — three gotchas:

<Callout type="warning">
  1. **`Published.toml`, not `published-at`.** This CLI tracks publish
  addresses in a separate `Published.toml` (`[published.testnet]` with
  `chain-id`, `published-at` = latest, `original-id`, `version`,
  `upgrade-capability`). There is **no `sui move manage-package`** subcommand
  in this build — seed `Published.toml` by hand. Remove any `published-at` from
  `Move.toml` (keeping both causes a "conflicting addresses" error).

  2. **CLI protocol < network → dry-run panics.** The local binary is protocol
  v124 while testnet is v126, so `--tx-digest` and the local compatibility
  check **panic**. Pass an explicit `--gas-budget` (skips the gas dry-run) and
  `--skip-verify-compatibility` (skips the local check; validators still
  enforce compatibility on-chain). A plain `publish` works without flags.

  3. **Type origin ≠ `original-id` in `Published.toml`.** The v4 upgrade was
  seeded with `original-id = 0xf2189af2…`, but the CLI resolves the true type
  origin (`0x6d968a48…`) from the on-chain package lineage via the UpgradeCap.
  So look up `AdminCap` / find owned caps by the **`0x6d968a48…`** type, not
  `CONFIG.PACKAGE_ID`.
</Callout>

## Deployed backend services

The hosted Render services the deployed frontend talks to:

| Service | URL | What it does |
|---|---|---|
| **Indexer** | `https://deepmarket-tusz.onrender.com` | Tracks `market_factory` events, persists fills + price history, serves `/markets`, `/markets/:id/history`, `/markets/:id/positions/:address`. |
| **Messaging relayer** | `https://sui-stack-messaging.onrender.com` | Sui Stack Messaging relayer — gRPC sub to Sui, encrypted message routing, `/health_check`. Built from a fork of MystenLabs/sui-stack-messaging. |
| **Telegram bot** | `https://deepmarketbot.onrender.com` | The notifier + autonomous trader + LLM agent. Telegram itself talks to the bot (via long-poll or webhook); the URL is mostly for `/health` pings and to make the service host-able. |

Both are on Render's free tier and pinged every 5 minutes by UptimeRobot to
defeat the 15-minute idle spin-down. First request after a long silence may
still take 30-60s to respond if the ping happens to land between sleep and
the next ping.
